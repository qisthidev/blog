│ Context                                                                      │
│                                                                              │
│ The pSEO content system is already built and working:                        │
│ - JSON datasets in src/data/pseo/ define structured content entries          │
│ - scripts/generate-pseo.ts converts JSON → Markdown blog posts               │
│ - scripts/validate-pseo.ts enforces quality guardrails (800+ words, code     │
│ blocks, FAQs)                                                                │
│                                                                              │
│ The bottleneck is manually authoring the JSON seed data. Each entry requires │
│  ~100 lines of structured JSON with root causes, symptoms, fix steps,        │
│ before/after code examples, metrics, and FAQs. This automation script uses   │
│ an LLM API to generate that structured data from a simple topic query.       │
│                                                                              │
│ What We Build                                                                │
│                                                                              │
│ File: scripts/automate-pseo.ts                                               │
│                                                                              │
│ A single CLI script that:                                                    │
│ 1. Accepts a topic query (e.g., "Go goroutine leak detection pprof") and a   │
│ content model type                                                           │
│ 2. Calls an LLM API (configurable: Anthropic, OpenAI, or Gemini) with a      │
│ structured prompt                                                            │
│ 3. Receives JSON matching the existing schema (ErrorGuide or                 │
│ PerformanceGuide)                                                            │
│ 4. Appends the new entry to the appropriate dataset file (errors.json or     │
│ performance.json)                                                            │
│ 5. Optionally creates a new hub dataset file if --new-hub is specified       │
│ 6. Runs generate-pseo.ts and validate-pseo.ts automatically                  │
│ 7. Reports success/failure                                                   │
│                                                                              │
│ Provider-Agnostic Design                                                     │
│                                                                              │
│ Uses a simple adapter pattern — one function per provider, all returning the │
│  same string (raw JSON). No heavy SDK dependencies; uses fetch() directly    │
│ against each provider's REST API.                                            │
│                                                                              │
│ PSEO_PROVIDER=anthropic|openai|gemini   (env var)                            │
│ PSEO_API_KEY=sk-...                      (env var)                           │
│ PSEO_MODEL=claude-sonnet-4-5-20250514|gpt-4o|gemini-2.0-flash  (env var,     │
│ optional)                                                                    │
│                                                                              │
│ CLI Interface                                                                │
│                                                                              │
│ # Generate a single entry and append to existing hub                         │
│ npx tsx scripts/automate-pseo.ts \                                           │

│ Structure:                                                                   │
│ 1. CLI arg parsing — parse --query, --model, --hub, --new-hub, --batch,      │
│ --dry-run, --no-generate                                                     │
│ 2. Environment loading — read PSEO_PROVIDER, PSEO_API_KEY, PSEO_MODEL from   │
│ env (or .env file)                                                           │
│ 3. Prompt construction — build a system prompt with the exact JSON schema    │
│ and quality requirements, plus the user's topic query                        │
│ 4. Provider adapters — callAnthropic(), callOpenAI(), callGemini() — each    │
│ takes a prompt, returns raw text                                             │
│ 5. JSON extraction & validation — parse the LLM response, extract JSON from  │
│ markdown fences if needed, validate against schema shape                     │
│ 6. Dataset mutation — read existing JSON file, append new entry, write back  │
│ 7. Pipeline execution — run generate-pseo.ts and validate-pseo.ts via        │
│ child_process                                                                │
│ 8. Reporting — print summary of what was generated, validation results       │
│                                                                              │
│ System Prompt Strategy                                                       │
│                                                                              │
│ The prompt includes:                                                         │
│ - The exact TypeScript interface for the target model (copied from           │
│ schema.ts)                                                                   │
│ - An example entry from the existing dataset (for tone/depth calibration)    │
│ - Quality constraints: root_cause must be 3+ sentences, symptoms must have   │
│ 5+ items, fix_steps must have 6+ items, code examples must be realistic and  │
│ commented, FAQs must have 2+ entries with 50+ word answers                   │
│ - A slug generation rule: kebab-case, technology-prefixed                    │
│ - Tag list from the hub's hub_tags to ensure consistency                     │
│ - Instruction to return ONLY valid JSON, no markdown wrapping                │
│                                                                              │
│ Provider Adapters                                                            │
│                                                                              │
│ All use native fetch() — no SDK dependencies needed:                         │
│                                                                              │
│ - Anthropic: POST https://api.anthropic.com/v1/messages with                 │
│ anthropic-version: 2023-06-01 header                                         │
│ - OpenAI: POST https://api.openai.com/v1/chat/completions with               │
│ response_format: { type: "json_object" }                                     │
│ - Gemini: POST https://generativelanguage.googleapis.com/v1beta/models/{mode │
│ l}:generateContent with responseMimeType: "application/json"                 │
│                                                                              │
│ Error Handling                                                               │
│                                                                              │
│ - Retry up to 2 times on network/rate-limit errors (429, 503)                │
│ - If JSON parsing fails, attempt to extract JSON from markdown code fences   │
│ - If schema validation fails, report which fields are missing/wrong          │
│ - Never silently corrupt existing dataset files — write to a temp file       │
│ first, then replace                                                          │

  New File: scripts/automate-pseo.ts

  A single CLI script that automates pSEO JSON seed data generation using LLM
  APIs. Key features:

  CLI Interface:
  # Single query into existing hub
  npx tsx scripts/automate-pseo.ts \
    --query="Go goroutine leak detection pprof" \
    --model=errors --hub=high-performance-laravel

  # New hub
  npx tsx scripts/automate-pseo.ts \
    --query="Docker multi-stage build Alpine PHP" \
    --model=performance --new-hub=modern-devops-ci-cd \
    --hub-title="Modern DevOps & CI/CD"
  --hub-tags="docker,devops-and-infrastructure"

  # Batch mode
  npx tsx scripts/automate-pseo.ts --batch=queries.txt --model=errors
  --hub=high-performance-laravel

  # Dry run (preview without writing)
  npx tsx scripts/automate-pseo.ts --query="..." --model=errors
  --hub=high-performance-laravel --dry-run

  Architecture:
  1. CLI arg parsing — --query, --model, --hub, --new-hub, --batch, --dry-run,
  --no-generate
  2. Env loading — reads PSEO_PROVIDER, PSEO_API_KEY, PSEO_MODEL from env or
  .env file
  3. Prompt construction — embeds the exact TypeScript schema with quality
  constraints (5+ symptoms, 6+ fix steps, 50+ word FAQ answers, etc.)
  4. Provider adapters — callAnthropic(), callOpenAI(), callGemini() using
  native fetch(), no SDK dependencies
  5. JSON extraction — handles raw JSON, markdown-fenced JSON, and brace
  extraction
  6. Schema validation — checks all required fields, array minimums, sentence
  counts, FAQ answer lengths
  7. Safe dataset mutation — writes to temp file first, then atomic rename;
  checks for duplicate slugs
  8. Pipeline execution — auto-runs generate-pseo.ts and validate-pseo.ts after
  appending
  9. Retry logic — retries up to 2 times on 429/503 with exponential backoff

  Modified: package.json

  Added "automate:pseo" script alongside existing generate:pseo and
  validate:pseo.

  No new dependencies — uses native fetch() and node:child_process.
